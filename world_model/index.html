Research era timeline

Specialist models -> foundation priors -> applied generation -> world models.

My path started with small, specialist models for digital humans, then moved into the foundation-model era: embracing large generative priors and bringing them back into downstream creative applications. As that paradigm became increasingly established, world models felt like the missing piece, because they force the hardest open problems in generation to be solved together: real-time synthesis, physical and causal consistency, long-context reasoning, and persistent 3D memory or world state.

WorldCam

WorldCam

Motivation

WorldCam is built around two requirements for interactive video worlds: more precise control and 3D world consistency.

Control

Users need a precise, continuous way to specify how the camera moves through a generated world.

Consistency

The world should remain spatially coherent when the camera moves, revisits, and reveals new regions.

Unified rep.

Through experiments, we found camera pose can serve as the unified representation for both motivations.

Camera pose for control + 3D consistency CVPRW 2026

Model Architecture

WorldCam converts user actions into camera poses in Lie algebra and conditions a progressive autoregressive video transformer on these camera poses for precise action control. Retrieved long-term memory latents and camera poses from the memory pool enforce 3D consistency of the generated world, while short-term memory with an attention sink stabilizes long-horizon generation.

The model needs both long-term and short-term memory; camera pose conditioning is the bridge that connects action control with memory-grounded 3D consistency.

Precise Action Control over Long-Horizon Generation

WorldCam enables precise action control through complex, entangled discrete keyboard and continuous mouse inputs.

3D World Consistency over Long-Horizon Generation

WorldCam preserves the underlying 3D scene structure during long-horizon rollouts, so when an agent revisits the same location from the same view angle, the scene remains recognizable and spatially consistent.

Future directions

Across my world-model work and other concurrent efforts in the field, the next stage is not one algorithm, but a coupled stack where stability, control, state, and speed all have to hold at once.

Future direction 1 Drifting

We have made progress on error accumulation, identity drift, and broken revisiting, but it remains a stress test for long horizons.

Future direction 2 Action control

WorldCam moves this forward with precise camera-pose control; the next challenge is broader action interfaces.

Future direction 3

World state

A representation that records what happened, what changed, and what can be revisited later.

Future direction 4 Realtime distillation

Turn high-quality world generation into a true interactive system with end-to-end low latency.

Active focus

World State

Two active internship projects are exploring world-state design: one learns compact memory for compression and registration, while the other builds shared state for multiplayer synchronization.

01

Latent state learning

Use an encoder to learn compact world-state representations, turning raw memory retrieval into latent retrieval.

Current frame-based retrieval is often dominated by heuristic rules and can miss the flexible memory needed for the current query. A learned representation should make retrieval more compact, adaptive, and robust.

02

Multiplayer sync

Use a shared world state plus cross-player synchronization so multiple players see one consistent evolving world, scaling toward arbitrary numbers of players, views, and actions.

World state as memory

World state is the map underneath a world model: it records what each place looks like, what happened there, what changed, and what should be remembered so long-horizon generation can revisit the same world with consistent place, event, and state.

Learned state

Model-native memory

The world model generates, compresses, stores, and retrieves its own latent memory.

Engine-backed state

External world host

A 3D, game, or physics engine maintains geometry, physics, state, and metadata.

Video model renderer

Same renderer, different state source

Both world-state forms are viable: they differ in whether state is generated and remembered by the model itself, or maintained by an external engine. In both paths, video models render rich realistic or stylized worlds from that state.

Future focus

Real-Time

Real-time generation is the key that turns a video world model into an interactive world model: actions only feel grounded when perception, control, memory, and rendering close the loop at human interaction speed.

Why it matters

Interaction needs latency, not just quality.

A world model becomes useful when the user can act, observe, correct, and act again. That requires long-horizon consistency, controllability, and few-step rendering to work together.

Why Roblox?

Roblox is already a world platform.

Video world models become most useful when they are grounded in persistent, interactive, multiplayer worlds. Roblox already has the external world host: game state, physics, avatars, creators, and social interaction. A video world model can become the generative renderer and real-time interface on top of that state.

external world hostvideo model renderer

The most useful bridge for Roblox may be real-time style rendering: keep the existing 3D engine state, then use world models to render it into realistic, cinematic, or stylized visual worlds.
